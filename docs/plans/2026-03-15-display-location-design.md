# 多显示器显示位置设计

## 背景

当前像素猫默认跟随 Dock 所在显示器，这对单屏和多数多屏场景已经够用，但仍缺少一个显式的“显示在哪块屏幕上”的运行时控制入口。用户希望：

- 动态检测当前显示器数量。
- 当存在多个显示器时，托盘菜单提供“显示位置”二级菜单。
- 默认模式为“自动”，继续沿用现有“跟随 Dock”的行为。
- 手动选择某个显示器后：
  - 若 Dock 在该显示器上，像素猫仍贴 Dock 上沿。
  - 若 Dock 不在该显示器上，像素猫退化为显示在该显示器底边。
- 当前阶段不做持久化，只在本次运行内生效。

## 目标

在不破坏现有 Dock 跟随逻辑的前提下，为多显示器场景增加一个轻量、可动态刷新的显示位置选择能力。

## 非目标

- 不做设置页或配置文件持久化。
- 不做“为不同显示器分别记忆位置”的复杂策略。
- 不解决物理显示器重排、重命名、镜像模式下的长期稳定标识问题；本期仅保证本次运行内的合理行为。

## 方案对比

### 方案 A：仅在当前检测到多显示器时显示菜单

优点：单屏用户界面最干净。

缺点：插拔显示器时菜单结构会突然出现或消失，交互层级不稳定。

### 方案 B：始终显示菜单

优点：菜单结构稳定。

缺点：对于始终单屏的用户，长期存在一个无意义入口。

### 方案 C：本次运行内延迟显现并保持稳定（推荐）

规则：

- 应用启动时若只有一个显示器，则先隐藏“显示位置”菜单。
- 只要本次运行中曾检测到过多显示器，就显示该菜单。
- 一旦显示，本次运行内即使后来只剩一个显示器，也不再移除该菜单，而是退化为说明状态。

优点：

- 对纯单屏用户不打扰。
- 对会插拔显示器的用户避免菜单反复闪变。
- 与“仅运行期状态”天然匹配，实现成本低。

## 推荐设计

采用方案 C，并把“显示位置”抽象为一个仅运行期有效的状态机：

- `Auto`：自动跟随 Dock 所在显示器。
- `Specific(selection_id)`：手动指定显示器。

其中 `selection_id` 只在当前运行内使用，用于把菜单选择与当前显示器快照关联起来。

## 运行时行为

### 自动模式

继续沿用现有行为：

- 通过 Dock 几何与屏幕快照确定 Dock 当前所在显示器。
- 若 Dock 在底部，贴 Dock 上沿。
- 若 Dock 在左/右侧，则沿目标显示器可见区域底边活动。

### 手动模式

先解析用户选择的目标显示器，再根据 Dock 是否位于该显示器决定布局：

- Dock 在目标显示器上：复用现有 Dock 布局采样结果。
- Dock 不在目标显示器上：构造一个“无 Dock floor 模式”样本，令像素猫沿该显示器底边活动。

这样可以复用现有 `DockPlacementSample -> AppliedLayout` 链路，避免增加第二套窗口定位公式。

## 托盘菜单设计

新增“显示位置”菜单组：

- 父级菜单：`显示位置` / `Display Location`
- 子项：
  - `自动`
  - 检测出的显示器列表

显示器列表要求：

- 运行时动态刷新，不在启动时写死。
- 文案优先使用显示器名称；若存在多个同名显示器，追加序号区分。
- 只在当前会话的托盘菜单层内保存，不落盘。

当当前只剩一个显示器、但本次运行中曾经出现过多显示器时：

- 保留“显示位置”父菜单。
- 子菜单保留：
  - `自动`（若已回退则打勾）
  - 一个禁用说明项，例如“当前仅检测到一个显示器”。

## 拔掉显示器时的处理

若用户当前手动选择的显示器消失：

- 运行时状态立即回退到 `Auto`。
- 后续布局重新按 Dock 自动跟随计算。
- 下次打开托盘菜单时，`自动` 子项应显示为勾选状态。

## 代码结构

### `src/cat.rs`

负责：

- 暴露给托盘使用的显示器菜单快照能力。
- 在现有 Dock 采样链路中加入“手动指定显示器”的分支。
- 在目标显示器不含 Dock 时，构造 floor 模式样本。

### `src/tray.rs`

负责：

- 保存仅运行期的显示位置模式。
- 在右键弹出前动态刷新“显示位置”菜单。
- 处理用户点击 `自动` 或某块显示器时的状态切换。

### `src/i18n.rs`

负责新增最小必要菜单文案：

- `Display Location`
- `Automatic`
- `Only one display detected`（或等价中文）

## 测试策略

优先补纯逻辑测试，覆盖以下场景：

1. 自动模式仍跟随 Dock。
2. 手动模式且 Dock 在所选显示器上，布局与自动模式一致。
3. 手动模式且 Dock 不在所选显示器上，退化为目标显示器底边 floor 模式。
4. 目标显示器消失时回退到 `Auto`。
5. 托盘菜单可见性状态遵循“本次运行曾出现多显示器后保持可见”的规则。

## 风险与约束

- 若当前显示器稳定标识仅依赖几何信息，运行时重排显示器可能导致手动选择丢失；本期接受该回退为 `Auto` 的行为，但会尽量在当前实现中使用更稳定的运行时选择标识。
- 托盘菜单刷新与 `NSScreen` 采样都必须在主线程完成，不能把 AppKit 读取放到后台线程。
